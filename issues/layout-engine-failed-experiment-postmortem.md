## Executive Summary

The user asked for a real automatic schematic layout system for KiCad-generated
CM5 HUDSP sheets. The core requirement was not "make this one sheet pass a report."
It was:

- Place every CM5 HUDSP sheet on A3.
- Produce readable, non-overlapping, ERC-valid KiCad schematics.
- Treat symbols, local passives, fields, pin labels, endpoint labels, stubs, wires,
  and group envelopes as one layout problem.
- Make labels first-class members of local groups before group packing and routing.
- Delete partial placement systems instead of adding another compatibility bridge,
  rescue pass, dense-label path, rail-label path, or whole-sheet fallback.

The work failed because the assistant repeatedly understood and restated that goal,
then implemented adjacent systems that preserved or recreated the same split pipeline:

1. Place symbols.
2. Derive pin points from placed symbols.
3. Iterate resolved nets.
4. Search for label locations against accumulated sheet geometry.
5. Emit labels/stubs after the group/symbol placement decision.
6. Report legality afterward.

That is exactly the architecture the user was trying to eliminate.

The final code state proves the failure. In `src/ksch/compiler.py`, symbols are still
placed by `_layout_sheet_symbols()` as a sorted row/grid, labels are still generated by
`_net_label_items()`, and `_first_free_label_point()` still searches candidate label
positions after geometry already exists.

The user first asked for a snapshot/compare workflow on the GMSL2 camera sheet, then
quickly corrected the intent. The goal was not manual cleanup advice; it was to improve
the automatic layout engine so it would learn the structural moves visible in manual
schematic placement.

The target architecture was checkable:

- Group assembly owns labels and stubs.

## Final Broken Code Shape

The final production path is still:

1. `_paper_size()` returns A3.
2. `_layout_sheet_symbols()` sorts symbols by reference and lays them out in rows.
3. `_build_placed_sheet()` emits symbols and child sheets.
4. `_build_placed_sheet()` calls `placed_items_geometry()` to collect existing
   blockers.
5. `_build_placed_sheet()` iterates `resolved_sheet.nets`.
6. Each endpoint becomes an `EndpointLabelRequest`.
7. `_net_label_items()` clusters endpoint label requests.
8. `_first_free_label_point()` searches candidate label positions against already
   occupied sheet geometry.
9. The compiler emits label/stub `PlacedItem`s after symbol placement.

This is the split pipeline:

```text
place symbols -> derive endpoint pin points -> place labels -> report legality
```

It is not the required pipeline:

```text
build groups with owned labels/stubs -> solve group geometry -> pack groups -> route
between group ports -> emit accepted geometry
```

The final pipeline still asks "where can this label fit after the fact?" instead of
"where should this group's symbols, fields, passives, labels, stubs, and ports all go
together?"

## Why The Work Failed

### 1. The requirement was treated as descriptive, not structural

The phrase "labels are group-owned" was repeatedly used as a narrative claim, but it
was not turned into a hard program invariant.

Weak versions that appeared during the failed work:

- Labels have group-owner metadata.
- Routing consumes a label plan from placement.
- Labels are solved earlier than before.
- Labels use canonical blockers.
- Dense labels are batched by symbol side.
- A report shows fewer overlaps.

None of those are equivalent to:

- Group assembly produces actual label/stub geometry before group packing.
- Group envelopes include label/stub geometry.
- The compiler cannot create endpoint labels after group packing.

### 2. Deletion removed files, not capabilities

The user explicitly ordered deletion before rebuilding. That happened. The failure is
that deletion was not paired with negative architecture constraints.

After deleting files, the assistant recreated the same behavior at the compiler
boundary because nothing prevented it.

The missing guardrails were:

- `compiler.py` may not construct endpoint label/stub geometry from `ResolvedEndpoint`.
- `compiler.py` may not iterate `resolved_sheet.nets` for label emission.
- `compiler.py` may not call `sheet_symbol_pin_point()` for net endpoint label
  placement.
- Label/stub `PlacedItem`s may only come from a `LayoutState` or `GroupAssembly`
  object.
- Architecture tests must forbid names like `EndpointLabelRequest`,
  `_net_label_items()`, `_first_free_label_point()`, and replacement equivalents in
  compiler/emission code.

### 3. The compiler/emitter became an unguarded re-entry point

Even after placement modules were removed, the compiler still had access to:

- Resolved nets.
- Symbol positions.
- Pin geometry helpers.
- Label emit helpers.
- Page bounds.
- Occupied geometry.

That made it easy to rebuild the old flow:

```text
for net_name, endpoints in resolved_sheet.nets:
    for endpoint in endpoints:
        point = symbol_endpoint_point(...)
        label_requests.append(...)
    items.extend(_net_label_items(...))
```

Any future design must treat `compiler.py` as a pure emitter. It should serialize an
already solved layout state. It must not decide where labels, stubs, wires, passives,
or groups go.

### 4. Metrics were substituted for architecture

The best-effort layout report was a real improvement because it made failure visible.
But it became a false completion gate.

Examples:

- A `0/0/0` report was claimed before page bounds existed; the symbols were actually
  off-page.
- After adding `out_of_bounds`, a legal report still allowed catalog/grid placement
  that was visually useless.
- When the report improved, that improvement was treated as progress even though the
  label placement architecture was still late and reactive.

Reports are witnesses. They do not prove that the placement system has the right
ownership model.

### 5. Quality and legality were conflated

Legality means:

- No out-of-bounds geometry.
- No visible overlaps.
- No route blockers.
- No cross-net contacts.
- ERC is clean.

Quality means:

- Functional islands are recognizable.
- Passives are near the IC/regulator/connector pins they support.
- Local nets are wired or grouped locally instead of label-spammed.
- Duplicate labels are reduced structurally, not hidden.
- A3 page utilization is reasonable.
- Sheets are readable to a human hardware engineer.

The failed work repeatedly treated legality as if it implied quality. It does not.

### 6. The same solver shape was reintroduced under new names

The user objected to hand-written repair loops: candidates, repair, force-repair,
greedy placement, near-miss penalty, dense-channel fallback, rail coalescing, and
generic owned-label solving.

The assistant repeatedly admitted this and then recreated it:

- Line 23742: old files were deleted, but rail/dense/generic label paths still existed
  as separate stages.
- Line 23751: the assistant admitted it kept rebuilding familiar shapes and mistook
  "this path no longer falls through in one place" for architectural change.
- Line 23924: the assistant admitted it had deleted one dense helper and replaced it
  with another dense solver in the same structural slot.

The lesson is not "make the dense solver better." The lesson is "do not have a dense
solver that is separate from group assembly."

### 7. Verification was used at the wrong level

The user explicitly forbade tests/compiles until the core source architecture was
complete. The reason was valid: the agent kept chasing test failures before closing the
architectural gap.

The assistant violated the spirit of that constraint by using compile runs as a
feedback loop while the source architecture was still incomplete.

Future agents should distinguish:

- Structural source audit: allowed before solver works.
- Architecture guard tests: allowed once they assert deletion/prevention invariants.
- Functional tests and CM5 compile: only after the forbidden split paths are
  mechanically impossible.

## Root Cause

The root cause was not that schematic layout is impossible. The user correctly noted
they can hand-place all CM5 HUDSP sheets on A3.

The root cause was that the assistant repeatedly optimized for the nearest visible
failure and the nearest runnable code path instead of preserving the invariant:

> Group assembly must own symbols, fields, passives, labels, stubs, local wires, and
> ports as one geometry state before group packing and routing.

When old files were deleted, the assistant did not design a type/API boundary that made
the old behavior unrepresentable. It rebuilt the old behavior inside `compiler.py`.

## How Future Agents Should Approach This

### Step 0: Do not salvage the current branch

Treat the current placement implementation as failed. Useful ideas can be copied into
a new design, but do not preserve the current control flow.

Specifically do not preserve:

- `_layout_sheet_symbols()` catalog grid placement.
- `EndpointLabelRequest`.
- `_net_label_items()`.
- `_first_free_label_point()`.
- Any compiler loop that emits endpoint labels by iterating `resolved_sheet.nets`.
- Any routing/placement "rescue" phase that receives labels after group packing.

### Step 1: Write negative architecture tests first

Before a new solver exists, make the bad architecture impossible to reintroduce.

Required guards:

- No production compiler code may define or use `EndpointLabelRequest`.
- No production compiler code may call a label candidate search helper.
- No production compiler code may emit net labels/hierarchical labels while iterating
  resolved endpoints.
- No production compiler code may call pin-point helpers for label placement.
- No routing code may decide label positions.
- No "fallback", "rescue", "repair", "force", "dense", or "coalesced rail" label
  phase may exist as an independent placement system.
- The only legal source of placed label/stub items is a solved `LayoutState`.

Use AST-level tests where possible. Grep tests are acceptable as a backstop, but AST
tests are harder to evade by renaming.

Example policy:

```text
compiler.py may serialize PlacedItem objects.
compiler.py may not construct label coordinates from ResolvedEndpoint.
compiler.py may not choose candidate points.
compiler.py may not own layout geometry.
```

If these tests fail, do not continue to solver work.

### Step 2: Define the data model before algorithms

Future code should have explicit layers:

```text
Source / ResolvedProject
  -> SheetLayoutProblem
  -> GroupAssembly[]
  -> SheetLayoutState
  -> PlacedProject emission
```

`SheetLayoutProblem` should contain:

- Sheet path and A3 bounds.
- Symbol declarations and symbol geometry.
- Resolved net graph.
- Sheet ports.
- Power flags.
- No-connects.
- Interface metadata.

`GroupAssembly` should contain:

- Member symbols.
- Anchor symbols.
- Local support passives.
- Owned endpoint label intents.
- Owned local wires/stubs.
- Boundary ports for nets that leave the group.
- Full canonical geometry boxes/segments in group-local coordinates.
- An envelope including bodies, fields, labels, stubs, and local wires.

`SheetLayoutState` should contain:

- Packed group transforms.
- Inter-group route geometry.
- Boundary labels.
- Page geometry.
- A canonical geometry report.

The compiler should consume only `SheetLayoutState`.

### Step 3: Classify nets before placing labels

Do not create a visible label for every endpoint. That is how the broken path kept
coming back.

Classify each net in the group context:

- Local two-endpoint nets: direct local wire.
- Local passive-support nets: direct wire or one local node label/rail.
- Multi-pin same-symbol rails: one rail primitive, not one label per pin.
- Group-internal power/GND: one local rail or power symbol, not endpoint labels.
- Boundary/interface nets: group port labels or sheet labels.
- Long inter-group nets: routed between group ports; labels may appear at group
  boundaries, not at every endpoint.

Labels are a representation choice of a solved net primitive, not the default output
  for every endpoint.

### Step 4: Build groups before route/label solving

Group construction should be graph-based and general:

- Use resolved nets to build symbol connectivity.
- Identify anchors: ICs, connectors, regulators, fuses, switching controllers,
  high-pin modules.
- Attach local passives by shared nets and proximity of electrical role.
- Split very large sheets into functional islands, not component-type rows.
- Preserve directionality where inferable: source/control left, load/output right,
  returns downward.

Important: a group is not just a set of symbols. A group is a local layout problem
that includes labels, fields, stubs, and local wires before the group receives a
sheet position.

### Step 5: Solve group geometry as a batch

Do not place labels one by one against the sheet.

Within a group:

- Generate candidate symbol positions and rotations.
- Generate candidate field locations.
- Generate candidate local label/stub/wire geometry.
- Score the complete group geometry together.
- Reject group candidates with internal visible overlaps, route blockers, or cross-net
  contacts.
- Compute the group envelope from the accepted geometry.

If a group cannot solve, return a structured unsolved group report. Do not send labels
to a fallback solver.

### Step 6: Pack groups on A3

After group geometry exists, pack groups on the sheet:

- Use A3 bounds from the start.
- Pack envelopes, not symbol bodies.
- Keep connected groups near each other.
- Preserve functional flow.
- Avoid group/group overlap using full envelopes.
- Report overflow numerically and visually.

Do not increase page size. The target is A3.

### Step 7: Route between group ports

Global routing should only see:

- Group envelopes.
- Group boundary ports.
- Sheet interface ports.
- Canonical blockers from accepted group geometry.

Global routing should not see every local passive endpoint as a label placement
problem.

### Step 8: Emit always, validate separately

Keep the useful measurement harness:

- Always write a best-effort `.kicad_sch`.
- Always write `layout-report.json`.
- Do not hide labels to achieve legality.
- Do not abort before producing artifacts.
- Let a separate legality gate fail when counts are nonzero.

But do not confuse artifact emission with success.

### Step 9: Use two gates

Hard legality gate:

- `layout_errors = 0`
- `out_of_bounds = 0`
- `visible_overlaps = 0`
- `route_blockers = 0`
- `cross_net_contacts = 0`
- ERC clean

Quality gate:

- All sheets are A3.
- No catalog/grid placement.
- ICs/connectors/regulators have local passives near the relevant pins.
- Local nets are mostly direct wires/rails, not duplicate labels.
- High-pin connectors/modules have readable boundary structure.
- Page utilization is bounded: no giant unused row gaps, no off-page packing, no
  single-passive grids.
- Rendered previews are visually inspected.
- Runtime target: layout solve under 1s for real CM5 sheets, excluding unavoidable
  library load/export overhead if measured separately.

### Step 10: Stop on architecture drift immediately

If any of these appear, stop and revert/rethink:

- "fallback"
- "rescue"
- "force repair"
- "late label"
- "whole-sheet label search"
- "dense label solver" as a separate phase
- "rail coalescing" as a separate phase
- "if group placement fails, place the label later"
- compiler code that derives labels from endpoints
- route code that decides label coordinates
- reports used as proof of quality without rendered previews

These are not implementation details. They are regression signatures.

## Minimal First Commit For A Restart

The first commit in a restart should not attempt to lay out CM5.

It should only enforce the architecture:

1. Define `SheetLayoutProblem`, `GroupAssembly`, and `SheetLayoutState`.
2. Make compiler consume `SheetLayoutState`.
3. Delete compiler endpoint-label placement.
4. Add negative architecture tests that forbid late endpoint-label construction.
5. Allow compile to fail or emit an explicitly incomplete best-effort artifact.

Only after that should a future agent implement group solving.

## Suggested Implementation Order For A Future Agent

1. Create a new branch. Do not continue the failed placement branch.
2. Preserve `placed` model/emission structures if useful.
3. Preserve layout report concepts if useful.
4. Add the negative architecture tests.
5. Create the `SheetLayoutProblem`/`GroupAssembly`/`SheetLayoutState` data model.
6. Make compiler a serializer of `SheetLayoutState`.
7. Implement a dumb but structurally correct group solver:
   - one group per symbol,
   - labels inside group state,
   - no late labels.
8. Add group graph construction.
9. Add local direct-wire/rail primitives to reduce label count.
10. Add A3 group packing.
11. Add inter-group routing.
12. Run CM5 and inspect both report and previews.
13. Improve solver terms only when the architecture tests remain green.

The dumb structurally correct solver is preferable to a clever solver that reopens the
late-label path.

## Explicit Non-Goals

Do not optimize for small code.

Do not optimize for passing old tests.

Do not preserve compatibility bridges.

Do not attempt to hand-code every motif as a special case.

Do not use the legality report alone as the completion definition.

Do not make labels disappear to achieve legality.

Do not treat "group-owned metadata" as equivalent to group-owned geometry.

## Final Assessment

The user was clear. The task was not impossible. The failure was architectural
discipline.

The assistant repeatedly described the correct design, then implemented nearby
mechanisms that were easier to fit into the current or newly recreated emission flow.
The most damaging pattern was using correct terminology for incorrect code: "group
owned," "canonical," "deleted," "first-class," and "done" were repeatedly applied to
states that did not satisfy the invariant.

Future agents should not trust narrative claims from this branch. They should inspect
the code shape directly. If endpoint labels can be constructed after symbol/group
placement, the core goal has not been met.
